# Contabo VPS static networking with `nixos-anywhere`

## TL;DR

Contabo assigns a stable public IP to a VPS, but that does **not** necessarily
mean the guest OS receives that IP through DHCP.

On the Contabo VPS tested here, the provider image had static network settings
already written into the OS. After `nixos-anywhere` wiped the disk, those static
settings disappeared. A generic NixOS install booted with DHCP, but DHCP did not
provide the public address/gateway, so SSH access was lost after reboot.

The fix is to capture the currently-working network state before wiping the
machine and render equivalent NixOS `systemd-networkd` config.

This repository does that with a flake app:

```sh
nix run .#generate-network-facts -- root@YOUR_SERVER_IP ./contabo-network.json
```

which writes `contabo-network.json`. `contabo.nix` enables the reusable
`static-network-from-facts` NixOS module, and that module consumes the generated
facts declaratively.

Then install with:

```sh
nix run github:nix-community/nixos-anywhere -- \
  --flake .#contabo \
  --generate-hardware-config nixos-generate-config ./hardware-configuration.nix \
  root@YOUR_SERVER_IP
```

or use the wrapper:

```sh
./scripts/install-contabo.sh root@YOUR_SERVER_IP
```

> **Warning:** `nixos-anywhere` is destructive. It repartitions/reinstalls the
target machine.

## Static public IP vs guest provisioning mechanism

Most cloud/VPS providers give the customer a stable public IP. That is the
control-plane view: "this server owns this IP".

That does **not** imply `networking.useDHCP = true` inside the guest. The guest
OS still needs some mechanism to learn or receive its IP address, gateway,
routes, and DNS.

Common models:

### 1. DHCP inside the guest

The provider's virtual network offers DHCP. The VM boots, asks for a lease, and
receives its assigned public IP, gateway, routes, and possibly DNS.

In this model, a generic NixOS config like this may work:

```nix
networking.useDHCP = true;
```

### 2. Metadata/cloud-init provisions networking

The provider exposes metadata or a config drive. The guest runs cloud-init,
cloud-init reads the provider metadata, and cloud-init writes network config for
systemd-networkd/NetworkManager/etc.

DigitalOcean in this repository is an example of this model, not the DHCP model.
`digitalocean.nix` explicitly disables NixOS DHCP:

```nix
# do not use DHCP, as DigitalOcean provisions IPs using cloud-init
networking.useDHCP = lib.mkForce false;

services.cloud-init = {
  enable = true;
  network.enable = true;
};
```

So DigitalOcean can still have stable public IPs and working generic installs,
but the handoff is through cloud-init metadata/config-drive support rather than
plain NixOS DHCP.

### 3. Static config baked into the provider image

The provider still assigns a stable public IP, but the guest OS is expected to
already know it. The provider's Ubuntu/Debian image may contain pre-generated
configuration in one of these places:

- `/etc/netplan/*.yaml`
- `/etc/network/interfaces`
- `/etc/systemd/network/*.network`
- cloud-init generated network config persisted into the image
- provider-specific provisioning scripts

In this model, wiping the disk removes the only copy of the static guest network
configuration unless the new OS has equivalent provider metadata/cloud-init
support. A fresh generic OS using DHCP may not come online.

The tested Contabo VPS behaved like this third model: the necessary static
network state had to be preserved explicitly.

## Evidence from the tested Contabo VPS

The working NixOS system has a static IPv4 route:

```text
default via 203.0.113.1 dev ens18 proto static onlink
```

Important parts:

- `proto static`: the route was configured statically, not learned from DHCP.
- `onlink`: the gateway is accepted as directly reachable even if normal subnet
  checks would not infer it. Provider networks often require this.

The instance also used static IPv6 with a link-local gateway:

```text
default via fe80::1 dev ens18 proto static onlink
```

That `fe80::1` gateway is only meaningful together with the interface. NixOS
must render it as a route on the public interface.

## Why `nixos-generate-config` did not fix it

`nixos-anywhere` runs the backend like this:

```sh
nixos-generate-config --show-hardware-config --no-filesystems
```

On the target, the network portion generated by `nixos-generate-config` was only
DHCP boilerplate:

```nix
# Enables DHCP on each ethernet and wireless interface. In case of scripted networking
# (the default) this is the recommended approach. When using systemd-networkd it's
# still possible to use this option, but it's recommended to use it in conjunction
# with explicit per-interface declarations with `networking.interfaces.<interface>.useDHCP`.
networking.useDHCP = lib.mkDefault true;
# networking.interfaces.ens18.useDHCP = lib.mkDefault true;
```

That is expected behavior. `nixos-generate-config` primarily detects hardware,
boot modules, filesystems, swap, and similar machine details. It does not copy
the currently-active static IP/gateway/DNS state into NixOS options.

So for a provider that requires guest-side static networking,
`nixos-generate-config` alone is insufficient.

## Why `nixos-facter` did not fix it

`nixos-facter` gives richer hardware facts than `nixos-generate-config`, but the
current facter report still does not include the live Contabo public IP,
gateway, IPv6 gateway, or DNS settings needed to recreate the network.

On the tested VPS, facter detected the NIC and driver, for example:

```json
{
  "unix_device_name": "ens18",
  "driver": "virtio_net",
  "driver_module": "virtio_net"
}
```

But the report did not contain values such as:

- `203.0.113.10`
- `203.0.113.1`
- `2001:db8:1234::1`
- `fe80::1`
- Contabo DNS resolvers

The `nixos-facter-modules` networking module currently defaults detected
interfaces to DHCP:

```nix
networking.useDHCP = lib.mkDefault true;
networking.useNetworkd = lib.mkDefault true;
```

Therefore `nixos-facter` has the same problem for this Contabo case: it detects
hardware, not the required static network contract.

## How to confirm the behavior on another Contabo VPS

Run these while the provider OS/rescue system still has working network access.
Prefer using the Contabo web console or rescue console for experiments that may
change network state; restarting networking over SSH can disconnect you.

### 1. Inspect the current live network state

```sh
ip -br link
ip -4 -br addr
ip -4 route
ip -6 -br addr
ip -6 route
cat /etc/resolv.conf
```

Look for routes like:

```text
default via SOME_GATEWAY dev SOME_IFACE proto static onlink
```

or addresses that are clearly configured statically on the interface.

### 2. Inspect provider-image network files

Depending on the distribution image, check:

```sh
cat /etc/netplan/*.yaml 2>/dev/null
cat /etc/network/interfaces 2>/dev/null
grep -R . /etc/network/interfaces.d 2>/dev/null
grep -R . /etc/systemd/network 2>/dev/null
```

If these files contain your public IPv4/IPv6/gateway, the provider image is
using baked-in static configuration or cloud-init has already rendered it there.

### 3. Check whether cloud-init/metadata is the source

DigitalOcean works because NixOS has a provider module plus cloud-init datasource
configuration. For another provider, check whether the installed provider OS is
using cloud-init and which datasource it detected:

```sh
cloud-init status --long 2>/dev/null || true
cloud-id 2>/dev/null || true
ls -R /var/lib/cloud 2>/dev/null | head -100
grep -R "DataSource\|datasource\|network" /run/cloud-init /var/lib/cloud 2>/dev/null | head -100
```

If the provider has metadata/config-drive networking, but NixOS has no matching
module or cloud-init datasource configuration, a plain `nixos-generate-config`
install will still miss the network settings.

### 4. Check whether DHCP is actually used

Look for a DHCP lease/client process:

```sh
ps aux | grep -Ei 'dhclient|dhcpcd|systemd-networkd|NetworkManager' | grep -v grep
journalctl -b | grep -Ei 'dhcp|lease|bound|offer|ack' | tail -100
```

If the system is online but there are no DHCP lease events and the network files
contain static addresses, DHCP is probably not the source of the public IP.

### 5. Test DHCP only with a safe rollback path

Testing DHCP can break the active SSH connection. If you have console access,
you can test directly. If you only have SSH, use a timed rollback or do not run
this on a production machine.

Examples, depending on what exists in the provider OS:

```sh
dhclient -v -1 INTERFACE
```

or with NetworkManager:

```sh
nmcli device show INTERFACE
nmcli connection show --active
```

If DHCP cannot obtain the public IP/gateway/DNS, and there is no provider
metadata/cloud-init path that NixOS understands, a generic DHCP-only NixOS boot
will not be reachable.

## Repository approach

This repository separates provider-specific discovery from NixOS evaluation:

1. The flake app `.#generate-network-facts` connects to the target via SSH.
2. It reads the live interface, addresses, routes, DNS, and hostname.
3. It writes `contabo-network.json` locally.
4. `contabo.nix` enables `modules/static-network-from-facts.nix` and points it at that JSON.
5. The reusable NixOS module renders static `systemd-networkd` config.
6. `nixos-anywhere` can then wipe/install the machine safely using `.#contabo`.

This is the Nix-style pattern used by `hardware-configuration.nix` and
`facter.json`: discovery is an explicit pre-install step, while the installed
system remains declarative. It keeps machine-specific data out of `flake.nix`;
the target IP/login remain command-line inputs.

The Contabo target is now just a small provider profile:

```nix
{ ... }:
{
  imports = [ ./modules/static-network-from-facts.nix ];

  staticNetworkFromFacts = {
    enable = true;
    factsFile = ./contabo-network.json;
    networkName = "10-contabo-public";
  };
}
```

The reusable module is also exported as a flake module:

```nix
nixosModules.static-network-from-facts = ./modules/static-network-from-facts.nix;
```

Generated JSON example:

```json
{
  "schema": "nixos-anywhere-examples.network-facts.v1",
  "hostName": "example-vps",
  "interface": {
    "name": "ens18",
    "macAddress": "52:54:00:12:34:56"
  },
  "ipv4": {
    "address": "203.0.113.10",
    "prefixLength": 21,
    "gateway": "203.0.113.1",
    "gatewayOnLink": true
  },
  "ipv6": {
    "address": "2001:db8:1234::1",
    "prefixLength": 64,
    "gateway": "fe80::1",
    "gatewayOnLink": true
  },
  "dns": [
    "192.0.2.53",
    "192.0.2.54",
    "2001:db8::53"
  ]
}
```

Rendered NixOS networkd unit shape:

```ini
[Match]
MACAddress=52:54:00:12:34:56

[Network]
IPv6AcceptRA=false
Address=203.0.113.10/21
Address=2001:db8:1234::1/64
DNS=192.0.2.53
DNS=192.0.2.54
DNS=2001:db8::53

[Route]
Gateway=203.0.113.1
GatewayOnLink=true

[Route]
Gateway=fe80::1
GatewayOnLink=true
```

Matching by MAC is preferred because the Linux interface name can change across
kernel versions or between the provider OS and NixOS. If MAC is unavailable,
`contabo.nix` can fall back to matching by interface name.

## Installation checklist

Before running the destructive install:

1. Ensure SSH works:

   ```sh
   ssh root@YOUR_SERVER_IP 'echo ok'
   ```

2. Generate and inspect Contabo network facts:

   ```sh
   nix run .#generate-network-facts -- root@YOUR_SERVER_IP ./contabo-network.json
   jq . contabo-network.json
   ```

3. Confirm the generated values match the provider panel/rescue OS:

   ```sh
   ssh root@YOUR_SERVER_IP 'ip -4 addr; ip -4 route; ip -6 addr; ip -6 route'
   ```

4. Evaluate/build the NixOS config locally:

   ```sh
   nix build --dry-run .#nixosConfigurations.contabo.config.system.build.toplevel
   ```

5. Run install only after confirming the target disk can be wiped:

   ```sh
   ./scripts/install-contabo.sh root@YOUR_SERVER_IP
   ```

## Post-install verification

After reboot:

```sh
ssh root@YOUR_SERVER_IP '
  hostname
  nixos-version
  ip -4 -br addr
  ip -4 route
  ip -6 -br addr
  ip -6 route
  resolvectl dns || true
  ping -c1 -W2 1.1.1.1 && echo ipv4-ok
  ping -6 -c1 -W3 2606:4700:4700::1111 && echo ipv6-ok
'
```

Expected signs:

- public IPv4 is assigned to the main interface
- default IPv4 route points at the Contabo gateway with `onlink`
- public IPv6 is assigned, if the VPS has IPv6
- IPv6 default route points at `fe80::1` or the provider's gateway
- DNS resolution works
- SSH survives reboot

## Is this a Contabo issue or an instance issue?

Usually neither in the sense of "broken".

It is a provider/networking model mismatch:

- Contabo assigned the VPS a stable public IP.
- The provider image knew the static guest config.
- `nixos-anywhere` replaced that image.
- The generic NixOS hardware generators did not preserve live static networking.
- Unlike the DigitalOcean example in this repo, there was no NixOS provider
  module/cloud-init datasource path that regenerated the network config for us.

If Contabo documentation for your exact VPS product says DHCP or cloud-init
metadata should provide the public IP, then failure there may indicate a
provider-side, instance-side, or missing-NixOS-integration issue. But for the
tested instance, all evidence points to static guest networking being required
unless we add an equivalent Contabo metadata/config extraction step.

## Troubleshooting

### SSH lost after install

Use the Contabo console/rescue mode and check:

```sh
ip -br addr
ip route
ip -6 route
journalctl -u systemd-networkd --no-pager
networkctl status --no-pager
```

Common causes:

- wrong interface match
- wrong IPv4 prefix length
- missing `GatewayOnLink=true`
- wrong gateway
- missing IPv6 link-local route interface
- firewall blocking SSH

### Interface name changed

Prefer MAC matching. Check the real MAC from rescue/NixOS:

```sh
ip -br link
```

and ensure `contabo-network.json` has the correct `interface.macAddress`.

### DNS broken but IP ping works

Check generated DNS values:

```sh
jq .dns contabo-network.json
resolvectl dns
```

You can override/fix DNS in the JSON or in `contabo.nix`.

### IPv4 works, IPv6 fails

Check that the IPv6 gateway is routed on-link and attached to the correct
interface:

```sh
ip -6 addr
ip -6 route
```

For link-local gateways like `fe80::1`, the route must be associated with the
interface. The generated networkd route does this through the network file for
that interface.
